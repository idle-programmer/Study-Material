Cache Eviction policies?
->When cache is full → which item to remove?
| Policy | Logic                        | Use Case                 | Django/Redis Example            |
| ------ | ---------------------------- | ------------------------ | ------------------------------- |
| LRU    | Remove Least Recently Used   | Web pages, API responses | redis.Redis().setex() (default) |
| LFU    | Remove Least Frequently Used | Images, static assets    | Long-term trending data         |
| FIFO   | Remove First In              | Logs, sequential data    | Simple queues                   |
| LIFO   | Remove Last In               | Stack operations         | Undo history                    |

**Q: "LRU vs LFU?"**
A: "LRU for recency (user sessions, API cache). 
   LFU for frequency (popular articles). 
   LRU faster to implement, LFU better hit ratio long-term."

**Q: "Redis eviction?"** 
A: "Redis: allkeys-lru (default), maxmemory-policy allkeys-lfu, 
   volatile-ttl (time-based). Django-redis uses LRU."

**Q: "Django cache choice?"**
A: "LocMemCache: LRU (dev). Redis: allkeys-lru (prod). 
   Memcached: LRU. Set CACHE_TTL for volatile keys."

# If your server crashes/restarts on which redis is hosted, how will you retrive the data stored on redis?
-> Redis stores data in memory, so if a standalone Redis instance crashes, data may be lost. 
In production, I would use Redis persistence (RDB/AOF) and preferably Redis replication through AWS ElastiCache.
RDB (Redis Database Snapshot)
AOF (Append Only File)


# Why did you choose Redis for OTP storage and validation? Couldn't you achieve the same functionality using 
PostgreSQL by storing the OTP with a 5-minute expiration time? Why use Redis instead?
-> Yes, storing OTPs in PostgreSQL is absolutely possible. In fact, many applications do it. However, 
Redis is usually preferred because OTPs are temporary, high-frequency data. Redis is an in-memory data store 
with built-in key expiration, making reads and writes extremely fast while avoiding unnecessary load on the 
primary database.
